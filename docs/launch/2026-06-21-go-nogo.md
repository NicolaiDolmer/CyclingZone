## 1. Evidens-checkliste (gates, ikke beslutninger)

Hvert kriterium = et **målbart signal** + **hvor det kommer fra**. Gaten er en *læsning af tallene*, ikke en mavefornemmelse. Tærskler markeret **[EJER-BESLUTNING]** sættes af ejeren — stilladset leverer signalet og en kilde, ikke en grænse.

### G1 — Tour-kohortens retention (primært signal)

| Felt | Indhold |
|------|---------|
| **Signal** | D1 / D3 / D7 retention for den/de signup-uger der falder i Tourens 1. uge. |
| **Kilde** | `get_cohort_retention(p_weeks)` → `/admin/sprint-metrics` (`AdminSprintMetricsPage.jsx`). Returnerer pr. kohorte-uge: `cohort_size`, `d1/d3/d7_eligible`, `d1/d3/d7_returned`, `d1/d3/d7_pct`. |
| **Aflæsning** | Find den kohorte-uge hvis `cohort_week` = Tour-start-ugen. Brug `d7_pct` som hovedtal; `d1_pct`/`d3_pct` som tidlige indikatorer (D7 er først målbar 7 dage efter signup → kan være `NULL` i selve gate-ugen — se note). |
| **Tærskel** | TDF-launch-plan §"Succeskriterier 26/7" angiver **D7 ≥ 30 %** som 26/7-mål (ikke gate-tærsklen i sig selv). **[EJER-BESLUTNING: gate-tærskel for ~11/7]** — hvilken D1/D3 (og evt. D7) der udgør GO, og om en lille kohorte (lav `eligible`) overhovedet bærer beslutningen. Options nedenfor. |

**Note om D7-timing (vigtigt for gate-datoen):** RPC'ens eligibility-gate kræver `signup + 7 dage <= now()` før en bruger tæller i `d7_eligible`. Hvis gaten ligger ~11/7 og Tour-kohorten startede 4/7, er D7 først fuldt målbar ~11/7+. → I selve gate-ugen kan `d7_pct = NULL`/tynd. **[EJER-BESLUTNING: hvilket dag-mærke bærer gaten]**:
- **(A) Læn på D1+D3** (målbare i gate-ugen) + D7 som efter-validering. *Benefit:* beslutning kan tages til tiden ~11/7. *Cost:* D7 (det stærkeste stickiness-signal) er ikke fuldt med. *Alternativ:* (B).
- **(B) Skub gaten til når D7 for Tour-kohorten er fuldt målbar** (~14/7). *Benefit:* fuldt D7-grundlag. *Cost:* senere marketing-start, mindre af Touren tilbage. *Alternativ:* (A).

### 2a. Direkte RPC-kald (autoritativt — brug dette)

```sql
-- Kør som admin-JWT eller service_role (RPC gat'er på is_admin() OR service_role).
-- p_weeks = antal seneste signup-uger; vælg så Tour-start-ugen er dækket.
SELECT public.get_cohort_retention( [TODO: p_weeks, fx 4] );
-- → jsonb: { weeks, generated_at, cohorts: [ { cohort_week, cohort_size,
--            d1_eligible, d1_returned, d1_pct, d3_*, d7_* }, ... ] }
```

**Aflæsning til gaten:**
1. Find objektet i `cohorts[]` hvor `cohort_week` = `[TODO: Tour-start-ugens mandag, YYYY-MM-DD]`.
2. Læs `d1_pct`, `d3_pct`, `d7_pct` for den kohorte.
3. `*_pct = NULL` betyder "endnu ikke målbar" (`*_eligible = 0`), **ikke** 0 % — behandl som "—", ikke som fejl.
4. Hold `cohort_size` for øje: en lille kohorte gør pct'erne støjende. **[EJER-BESLUTNING: minimum `cohort_size` før pct tæller]**.

### 2b. Rå skeleton (kun hvis RPC'en skal reproduceres ad hoc — ellers SPRING OVER)

> Dette er en **forsimplet** illustration af RPC'ens logik for forståelse. Den autoritative definition (eligibility-gate, UTC-uge-trunkering, UNION af `last_seen` + `player_events`) bor i RPC'en — brug §2a i praksis.

```sql
-- ILLUSTRATIV — ikke til produktion. Verificerede kolonner:
--   auth.users.created_at · public.users.last_seen · player_events.user_id/created_at
WITH user_activity AS (
  SELECT
    au.id,
    au.created_at AS signup_at,
    GREATEST(
      COALESCE(pu.last_seen, au.created_at),
      COALESCE((SELECT max(pe.created_at) FROM public.player_events pe
                WHERE pe.user_id = au.id), au.created_at)
    ) AS last_activity
  FROM auth.users au
  LEFT JOIN public.users pu ON pu.id = au.id
  WHERE au.created_at >= [TODO: kohorte-cutoff, fx date_trunc('week', now()) - interval '3 weeks']
)
SELECT
  date_trunc('week', signup_at) AS cohort_week,
  count(*) AS cohort_size,
  -- D7 rolling: aktiv mindst 7 dage efter signup, kun talt hvis 7 dage er gået.
  count(*) FILTER (WHERE signup_at + interval '7 days' <= now())                                                       AS d7_eligible,
  count(*) FILTER (WHERE signup_at + interval '7 days' <= now() AND last_activity >= signup_at + interval '7 days')     AS d7_returned
  -- [TODO: tilføj D1/D3 efter samme mønster hvis ad hoc — ellers brug RPC'en]
FROM user_activity
GROUP BY date_trunc('week', signup_at)
ORDER BY cohort_week DESC;
```

**Bevidst IKKE opfundet:** ingen kolonner ud over de verificerede (`created_at`, `last_seen`, `user_id`). Funnel-event-navne (`race_viewed`/`training_focus_set`) er `[antagelse]` indtil bekræftet i `logEvent.js` — brug dem ikke i en gate-query før verifikation.

### 2c. Cross-check mod rullende aggregat

### 3b. Interview-guide (semistruktureret — tilpas pr. segment)

**Opvarmning (kontekst, ingen ledende spørgsmål):**
- Hvad fik dig til at prøve CyclingZone?
- Beskriv hvad du gjorde de første 10 minutter. Hvor gik du i stå?

**Kerne-loop (de fire motorer — løb · træning · ungdom · transfer/auktion):**
- Hvilken del trak dig tilbage næste dag? Hvilken ignorerede du?
- Auktioner: var budgivningen klar? Hvad forvirrede?
- Løb: gav etape-for-etape-afviklingen mening / var den værd at følge?
- Træning: forstod du hvad dine valg gjorde ved rytterne?
- Ungdom/udvikling: mærkede du progression over tid?

**Retention & deling (binder til gate-konsekvensen):**
- Hvad ville få dig til at komme tilbage hver dag? Hvad ville få dig til at stoppe?
- Ville du dele spillet? Med hvem, og hvorfor / hvorfor ikke? *(input til viral-loop #1173)*

**Afslutning:**
- Én ting du ville ændre i morgen?
- Må jeg vende tilbage med opfølgning?
