# Load-test: launch-kapacitet før marketing (#1174)

> **Dato:** 2026-06-11 (Europe/Copenhagen) · **Kontekst:** TdF-relaunch 20/6, marketing tændes derefter (Fase 1→2, jf. `docs/superpowers/specs/2026-06-09-tdf-validation-roadmap-design.md` §7).
> **Hård regel overholdt:** INGEN trafik mod prod. Al måling er lokal (backend + mock-Supabase på loopback). Prod er kun læst read-only (auth-statistik + DB-settings via Supabase MCP).

## TL;DR

**Backenden er IKKE flaskehalsen — det er Supabase Auths e-mail-kæde.**

- **MÅLT:** Backendens signup-bootstrap-kæde klarer **≥300 signups/sekund** lokalt (p95 < 270 ms ved 100 samtidige, 0 % fejl). Aktiv-bruger-trafik (dashboard-read + presence-pulse): **≥500 req/s**.
- **HÅRD GRÆNSE (dokumenteret):** E-mail-bekræftelse blev slået TIL ~10/6 (verificeret i `auth.users`: alle signups efter 10/6 har `confirmation_sent_at`; alle før 30/5 var autoconfirm). Dermed gælder Supabases **projekt-globale e-mail-rate-limit**: **2 mails/time** med indbygget SMTP, **30/time** default med custom SMTP (kan hæves).
- **Konklusion:** Systemet tåler i værste fald **~2 bekræftede signups/time** i dag (hvis custom SMTP ikke er sat op — UAFKLARET, skal verificeres i dashboardet), ellers **~30/time**. Efter custom SMTP + hævet limit: **~1.500+ signups/time** (næste flaskehals: per-IP-limit 30/5 min ved CGNAT/mobilnet). Backend/DB/Railway når først grænsen ved **titusinder pr. time**.

## 1. Metode + miljø

### Valg: lokal backend + mock-Supabase + analytiske eksterne grænser (option c)

Begrundelse for fravalg af de andre muligheder:

- **Supabase-branch (option a):** En branch kører GoTrues ÆGTE per-IP-rate-limits — en load-test fra én maskine ville måle rate-limiteren (30 signups/5 min/IP), ikke kapaciteten. Marketing-trafik kommer fra mange IP'er, så per-IP-grænsen er ikke den aggregerede flaskehals. Branchen ville altså give et tal vi allerede kender fra dokumentationen.
- **Lokal supabase CLI-stack (option b):** Docker-overhead på Windows + lokal GoTrue har andre rate-limit-defaults end hosted — hverken hurtigere eller mere retvisende end (c).
- **Valgt (option c):** Backend kørt lokalt (uændret produktion-kode) mod en mock af GoTrue + PostgREST med **injiceret netværkslatens** (10 ms/kald ≈ Railway europe-west ↔ Supabase eu-central-1 RTT). Det måler backendens reelle kæde-throughput; de eksterne Supabase-grænser er dokumenterede tal og lægges oven på analytisk.

### Miljø (maskeret)

| Komponent | Værdi |
|---|---|
| Backend | `backend/server.js` uændret, Node v24, port 3101, rate-limiters AKTIVE |
| Supabase-mock | `scripts/loadtest/mock-supabase.js`, loopback :54399, 10 ms injiceret latens pr. kald (0 ms-kontrolkørsel også udført) |
| Secrets | Ingen — backend bootet med dummy `SUPABASE_SERVICE_KEY`; mock accepterer syntetiske tokens (`lt-<n>`) |
| Driver | `scripts/loadtest/run-loadtest.js` — N workers, global bruger-tæller, p50/p95/p99 + fejlrate pr. niveau |
| Testmaskine | Windows 11, driver+backend+mock på samme maskine (CPU-kontention ⇒ tallene er **nedre grænser**) |

### Signup-kæden (kortlagt fra kode)

Fuldt signup-flow for en ny spiller:

1. **Frontend → Supabase PostgREST (anon):** holdnavne-precheck (`teams ilike`) — `LoginPage.jsx:118`
2. **Frontend → Supabase Auth:** `POST /auth/v1/signup` (GoTrue: bcrypt-hash + insert i `auth.users` + DB-trigger `handle_new_user()` → `public.users` + **bekræftelses-mail** siden ~10/6)
3. Bruger klikker mail-link → `/auth/v1/verify` → session
4. **Frontend → backend:** `PUT /api/teams/my` (bootstrap; `LoginPage.jsx:154` / `SetupWizardModal.jsx:38`) → `requireAuth` (GET `/auth/v1/user` + teams-lookup) → navnetjek → divisions-scan (`pickDivisionForNewTeam` læser ALLE rigtige hold) → team-insert → board-profil-insert = **~7 sekventielle Supabase-roundtrips**

Load-testen måler trin 4 (hele backend-kæden) direkte; trin 2-3 er Supabase-hostede og dækkes af de dokumenterede grænser i §3.

### Eksisterende backend-rate-limits (`backend/lib/rateLimiters.js`)

In-process, nøgle = user-id (fallback IP). Signup-bootstrap rammer `marketWriteLimiter` (30/min **pr. bruger**) — nye brugere har hver sin nøgle, så den begrænser ikke aggregeret signup-throughput. Presence/reads: 120/min pr. bruger. **OBS:** in-memory store ⇒ kun gyldig med 1 Railway-replica (kendt, jf. #1182) — skalér OP, ikke UD.

## 2. Måleresultater (MÅLT, lokalt)

### Scenario A — signup-bootstrap (`PUT /api/teams/my`), 10 ms injiceret Supabase-latens

300 signups pr. niveau, unikke brugere/holdnavne, rate-limiters aktive:

| Samtidighed | Throughput (signups/s) | p50 (ms) | p95 (ms) | p99 (ms) | Fejlrate |
|---|---|---|---|---|---|
| 5 | 45 | 109 | 122 | 129 | 0 % |
| 10 | 89 | 109 | 133 | 139 | 0 % |
| 25 | 222 | 109 | 127 | 133 | 0 % |
| 50 | 418 | 113 | 126 | 128 | 0 % |
| 100 | 483 | 192 | 247 | 266 | 0 % |

Lineær skalering til ~50 samtidige; ved 100 begynder kø-dannelse (p50 109→192 ms) men stadig 0 % fejl. Kontrolkørsel med 0 ms latens: 242-321 signups/s (p95 123-421 ms) — her er testriggens egen CPU-kontention (3 processer på én maskine) målbart loftet, ikke backenden alene.

### Scenario B — aktive brugere (`GET /api/me/onboarding-progress` + `POST /api/presence`)

Round-robin over 1.500 oprettede brugere (pr-bruger-rate holdt under presence-limiteren):

| Samtidighed | Throughput (req/s) | p50 (ms) | p95 (ms) | p99 (ms) | Fejlrate |
|---|---|---|---|---|---|
| 5 | 104 | 47 | 57 | 62 | 0 % |
| 10 | 197 | 49 | 64 | 68 | 0 % |
| 25 | 399 | 60 | 83 | 93 | 0 % |
| 50 | 562 | 87 | 121 | 135 | 0 % |
| 100 | 520 | 181 | 285 | 319 | 0 % |

Mætning ~550 req/s. Til sammenligning genererer **100 aktive managers** (9/6-audittens baseline, CPU-peak 4,3 % i prod-tomgang) i størrelsesordenen **1-5 req/s** — backenden har >100× headroom på aktiv-bruger-trafik.

### Race-conditions fundet og empirisk bekræftet (MÅLT)

1. **Duplikerede holdnavne:** 20 samtidige signups med SAMME holdnavn → **3×201 Created** + 17×409. `teams.name` har ingen UNIQUE-constraint (`database/supabase_setup.sql:27`); navnetjekket i `teamProfileEngine.js` er check-then-insert. Under marketing-burst får flere hold samme navn.
2. **Dobbelt-bootstrap samme bruger (alvorligere):** 5 samtidige `PUT /api/teams/my` for samme nye bruger → **2×201** = to hold til én bruger. Derefter fejler `requireAuth`s `.single()`-lookup stille → kontoen fremstår hold-løs (dashboard viser "team_named: false") selvom 2 hold findes. `teams.user_id` mangler UNIQUE-constraint. Udløses af dobbeltklik/to faner/netværks-retry i signup-øjeblikket.
3. **Divisions-kapacitet (analytisk, ikke testet isoleret):** `pickDivisionForNewTeam` er check-then-insert på divisions-tælling — samtidige signups kan overfylde en division (20→21+). Selvhelende ved næste sæson-transition, lav alvorlighed.
4. **Bonus-fund (død route):** `GET /api/teams/my` skygges af `GET /teams/:id` (registreret først, `api.js:3107` vs `:3125`) — `:id="my"` giver 404 i prod. Ingen frontend-kald rammer den i dag; bør ryddes op.

## 3. Eksterne hårde grænser (DOKUMENTERET — den reelle flaskehals)

### Supabase Auth ([rate-limits-doc](https://supabase.com/docs/guides/auth/rate-limits), defaults fra [supabase/shared-data/config.ts](https://github.com/supabase/supabase/blob/master/packages/shared-data/config.ts))

| Grænse | Default | Scope | Konfigurerbar? |
|---|---|---|---|
| **E-mails (signup/recover/email-skift)** | **2/time** (indbygget SMTP) · **30/time** (custom SMTP-default) | **HELE projektet** | Kun med custom SMTP — så frit ([doc](https://supabase.com/docs/guides/auth/auth-smtp)) |
| Sign-ups + sign-ins | 30 pr. 5 min | pr. IP | Ja (dashboard) |
| Verifications (`/auth/v1/verify`) | 360/time (burst 30) | pr. IP | Nej |
| Token-refresh | 1.800/time (burst 30) | pr. IP | Nej |
| OTP | 30/time | projekt | Ja |

**Kritisk kæde:** E-mail-bekræftelse er slået TIL i prod (verificeret read-only 11/6: alle 5 signups 10/6 fik `confirmation_sent_at`, bekræftet efter 31-84 s; alle signups ≤30/5 var autoconfirm med 0 s delay). Hver ny spiller koster altså én mail af det projekt-globale budget:

- **Indbygget SMTP:** 2 mails/time, og [siden sept. 2024 leveres kun til projekt-teammedlemmer](https://supabase.com/docs/guides/auth/auth-smtp) → reelt **dødt stop** for marketing.
- **Custom SMTP:** default 30 mails/time, kan hæves vilkårligt i dashboardet.
- At mails faktisk blev leveret til beta-brugere 10/6 *tyder* på at custom SMTP allerede er sat op — men volumen (max 5/dag, aldrig 2 inden for én time) kan ikke skelne de to. **SKAL verificeres i dashboardet (Authentication → Emails/SMTP) — 2 minutter.**

**Per-IP-fælden ved TdF-marketing:** Mobiltrafik bag carrier-NAT (CGNAT) deler IP'er. 30 signups/5 min pr. IP kan ramme falsk-positivt hvis et opslag går viralt på mobilnet. Grænsen er konfigurerbar — hæv den før kampagner.

### Supabase database (Pro-plan siden 10/6, #1181)

Read-only-verificeret 11/6: `max_connections=60`, `shared_buffers=224MB` ⇒ **Micro-compute** (Pro-default). Backenden bruger PostgREST over HTTP (ingen direkte pool) — 31/60 connections optaget af platformens egne services i tomgang. Signup-skrivningerne (3 inserts + 3-4 selects) er trivielle for Postgres; PostgREST på Micro klarer erfaringsmæssigt flere hundrede simple queries/s. GoTrue (bcrypt ~100 ms CPU pr. signup på delt core) estimeres til **~10-20 signups/s** — stadig 1.000+ gange over e-mail-grænsen. Opgradér compute (Small/Medium) først hvis Tour-trafik materialiserer sig; det er et klik, ikke en arkitekturændring.

### Railway ([plan-doc](https://docs.railway.com/reference/pricing/plans))

Ingen HTTP-rate-limits på services; grænser er ressourcer (Hobby: op til 48 vCPU/48 GB inkl. replicas; egress $0,05/GB). Backendens målte ceiling (~300-500 signups/s på én Node-proces) ligger langt under planloftet. 9/6-baseline: CPU-peak 4,3 % ved 100 aktive managers.

## 4. Kapacitetskonklusion

| Led i kæden | Kapacitet | Kilde |
|---|---|---|
| Backend signup-bootstrap | **≥ 300 signups/s** (≈ 1.000.000/time) | MÅLT (nedre grænse, lokal rig) |
| Backend aktiv-bruger-trafik | **≥ 500 req/s** ≈ flere tusinde samtidige aktive | MÅLT |
| Supabase GoTrue + DB (Micro) | ~10-20 signups/s ≈ 36.000+/time | ESTIMERET |
| Supabase e-mail (custom SMTP, default) | **30 signups/time** | DOKUMENTERET |
| Supabase e-mail (indbygget SMTP) | **2/time, kun teammedlemmer** | DOKUMENTERET |
| Per-IP signup-limit | 30/5 min pr. IP (CGNAT-risiko) | DOKUMENTERET |

> **Systemet tåler i dag ~30 bekræftede signups/time (worst case 2/time hvis custom SMTP mangler); første flaskehals er Supabase Auths projekt-globale e-mail-rate-limit — IKKE backend, DB eller Railway, som først mætter ved titusinder signups/time. Samtidige aktive brugere: 1.000+ uden sved (måling: ≥500 req/s mod et behov på ~1-5 req/s pr. 100 managers).**

## 5. Anbefalede tiltag før 20/6 (prioriteret)

1. ✅ **UDFØRT 11/6 — Custom SMTP opsat:** afklaring viste at custom SMTP IKKE var sat op (reel kapacitet var altså 2/time). Ejer opsatte Resend samme dag: domæne `cyclingzone.org` tilknyttet via Vercel-integration (DKIM/SPF/MX verificeret live i DNS), sending-only API-nøgle i Infisical dev (`RESEND_API_KEY`), SMTP-felter i dashboardet (smtp.resend.com:465, afsender `noreply@cyclingzone.org`). **End-to-end-verificeret** med ejer-godkendt test-signup mod prod: HTTP 200 + `confirmation_sent_at` sat (GoTrue ville have fejlet med 500 ved SMTP-afvisning); testbruger slettet bagefter. OBS: Resend gratis-tier = 100 mails/dag → opgradér til Pro ($20/md) før kampagne-peak hvis >100 signups/dag forventes.
2. ✅ **UDFØRT 11/6 — limits hævet af ejer:** e-mail-rate-limit ~500/time + signup-per-IP 100/5 min (dashboard). Nød-håndtaget (autoconfirm) består som launch-dag-fallback.
3. ✅ **UDFØRT 11/6 — UNIQUE-constraints live i prod** ([#1264](https://github.com/NicolaiDolmer/CyclingZone/issues/1264) lukket via PR #1273): `teams_user_id_unique_idx` + `teams_name_lower_unique_idx` (begge partial) verificeret via `pg_indexes`; idempotent 23505-håndtering i `teamProfileEngine.js`; backwards-check 0 dubletter.
4. ✅ **UDFØRT 11/6 — død route fjernet** (samme PR #1273).
5. **🟢 Efter launch:** Supabase compute Micro→Small hvis Tour-trafikken lander; Railway-replica-skalering kræver først delt rate-limit-store (#1182-noten).

## 6. Reproduktion

```bash
# Terminal 1 — mock (10 ms simuleret Supabase-RTT)
LT_LATENCY_MS=10 node scripts/loadtest/mock-supabase.js

# Terminal 2 — backend mod mock (ingen secrets)
cd backend && PORT=3101 SUPABASE_URL=http://127.0.0.1:54399 \
  SUPABASE_SERVICE_KEY=local-loadtest-dummy node server.js

# Terminal 3 — ramp
node scripts/loadtest/run-loadtest.js --base http://127.0.0.1:3101 \
  --levels 5,10,25,50,100 --per-level 300 --scenario both
```
